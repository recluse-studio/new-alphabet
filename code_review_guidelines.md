New Alphabet should be reviewed as a system, not as a collection of isolated files. The main question in every review is whether the change strengthens the framework’s constitution: one source of truth, narrow APIs, explicit states, stable naming, predictable composition, and low drift between philosophy, schema, implementation, and documentation. Code should be accepted when it makes the system more legible, more deterministic, and easier for both humans and agents to reason about. Code should be pushed back when it adds hidden rules, duplicate truth, ad hoc exceptions, overly broad APIs, or styling freedom that bypasses the framework’s grammar.

At the architectural level, every rule should exist in one canonical place. Data should not be restated across enums, match blocks, schemas, docs, and CLI logic unless there is a deliberate generated relationship between them. Prefer declared tables, typed configuration, and shared contracts over repeated imperative logic. Keep boundaries between foundation, primitives, semantic components, recipes, CLI, and schema layers very clear. A recipe must not smuggle in foundation logic. A component must not invent spacing outside the token system. The CLI and agent-facing surfaces must not quietly reimplement framework rules from scratch.

At the API level, favor small, typed, explicit interfaces. Every public item should have a narrow purpose, a stable name, and obvious constraints. Avoid prop or parameter surfaces that attempt to handle every future case. Prefer a few strong variants over many weak toggles. Unsupported combinations should be impossible, or rejected clearly and early. Defaults should be deliberate and documented. The shape of the code should make misuse difficult.

At the implementation level, prefer declarative structures over repeated branching logic whenever the code is expressing framework rules. If the same mapping, variant, token, or state appears in multiple places, treat that as a maintenance smell and look for a single source of truth. Keep logic total and exhaustive. Avoid hidden fallbacks, magical defaults, or “catch-all” behavior in core framework paths. New Alphabet should read like a governed system, not a chain of local improvisations.

At the design-system level, every change should be tested against the framework grammar. Does it align to the grid. Does it use official spacing tokens. Does it preserve hierarchy through type and rhythm instead of ad hoc decoration. Does it fit an existing primitive or recipe model. Does it create a new exception that weakens the larger language. The bar is not whether a screen “looks fine.” The bar is whether the change remains constitutional.

State modeling should be first-class everywhere. Any meaningful component or recipe should explicitly account for default, loading, empty, error, success, and disabled states where relevant, plus focus and keyboard behavior for interactive elements. Dense or data-heavy surfaces should also define truncation, overflow, wrapping, and zero-result behavior. No component should ship with unstated or accidental states.

Accessibility should be reviewed as a structural concern, not a final polish pass. Require semantic markup where appropriate, clear accessible names, keyboard reachability, visible focus treatment, and non-color-only meaning. Reviewers should ask whether the component remains understandable and operable under real constraints, not just whether it renders attractively.

Testing should follow the same philosophy as the code: test the rules, not just the happy path. Every foundational mapping should have deterministic tests. Every public component should have state coverage. Every recipe should have integration coverage. The repo should include regression tests that make system changes obvious in review. Good repo-wide expectations are: unit tests for core rules and contracts, integration tests for composed behavior, conformance tests for schema parity and anti-patterns, accessibility checks for interactive flows, and snapshot or visual regression tests for key recipes. When a rule is meant to remain stable, write a test that freezes it. When a rule is meant to evolve safely, write a test that makes the change explicit.

Security should be treated as disciplined predictability, especially because this repo includes framework code, CLI behavior, schema export, and future agent-facing surfaces. The main risks are not only classic exploits, but also unsafe file writes, unvalidated inputs, hidden execution paths, dependency drift, and silent contract mismatch. Reviewers should require explicit handling of all file-system operations, clear validation of all external or user-provided data, no unchecked assumptions when parsing or exporting schemas, and no hidden network or command execution in normal framework paths. Keep trusted and untrusted inputs clearly separated. Reject unknown schema fields or unsupported values explicitly rather than silently accepting them. Avoid panic-prone behavior in CLI and export flows when invalid input can be handled as an error.

For repo-wide security hygiene, require dependency review, locked versions where appropriate, linting, formatting, and automated checks in CI. Unsafe Rust should be treated as exceptional and justified in writing. Any use of unsafe should come with a narrow explanation, tests, and a clear reason it cannot be avoided. Secrets, tokens, and local environment assumptions should never be baked into examples, scripts, or docs. Example projects should be safe to clone and run without hidden side effects.

Documentation should be reviewed as part of the implementation, not after it. If a change affects the framework grammar, public APIs, recipes, CLI behavior, schema shape, or agent workflow, the docs should move with it. The code, schemas, docs, and examples should tell the same story. Drift between those layers is a real defect.

Because this project is AI-first, every review should also ask whether the change makes the system easier or harder for an agent to use correctly. Good changes make primitives more discoverable, rules more explicit, naming more stable, and schemas more complete. Bad changes force inference, rely on vibes, or bury important constraints in scattered implementation details. An agent should be able to inspect the repo and understand what is allowed, what is required, and what is forbidden without guessing.

The simplest summary standard is this:

Accept changes that reduce ambiguity, reduce duplication, increase explicitness, preserve layer boundaries, strengthen testing, and keep the framework legible as a single governed language.

Push back on changes that add hidden rules, broaden APIs unnecessarily, duplicate truth, weaken state handling, bypass the grammar, or make the repo harder for either humans or agents to reason about.
